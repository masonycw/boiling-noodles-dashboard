## 🚀 Phase 1: 基礎工程與中轉站建置
- [x] **Step 1: 設定 GCP 主機 2GB Swap 虛擬記憶體** (防止主機當機)
- [x] **Step 2: 在 GCP 主機上安裝 PostgreSQL**
- [x] **Step 3: 調整 PostgreSQL 記憶體設定檔** (降低 `shared_buffers` 適應 2GB RAM 環境)
- [x] **Step 4: 建立資料庫與連線專屬帳號** (`dashboard_user`)
- [x] **Step 5: 設計核心資料表 Schema**
    - `orders_fact` (所有訂單明細)
    - `order_details_fact` (品項明細)
    - `daily_revenue_agg` (每日營收彙總)
    - `member_profile_agg` (會員輪廓)
    - `data_freshness` (資料新鮮度追蹤)
- [x] **Step 6: 撰寫 `data_pipeline.py` (ETL) 腳本** — 歷史資料首次清洗與匯入
- [x] **Step 7: 部署 `file_watcher.py`** — 偵測到新資料檔案後自動觸發 ETL
    - 支援格式：`.csv`, `.xlsx`, `.xls`, `.json`, `.txt`（含 Eats365 API JSON）
    - 穩定延遲：10 秒（確認檔案上傳完成後才觸發）
- [x] **Step 8: 增量更新與系統防護 (Incremental ETL & Singleton Lock)** — **已完成**
    - **解決問題：** `data_pipeline.py` 每次掃描需 10~20 分鐘，且多次觸發會造成 DB Deadlock 鎖死。
    - **實作方式：**
        1. 導入 **單例鎖守門員** (`/tmp/pipeline.lock`)，確保 ETL 程序不會並發卡死。
        2. 使用 `processed_files.json` 記錄歷史檔案修改時間 (`mtime`)。
        3. `data_loader.py` 只解析新進或異動檔案，並由 `data_pipeline.py` 透過 `UPSERT` 寫回 DB。更新時間由 10 分鐘縮短至 **1秒內**。
